# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Typograph (published as `@overstacked/typograph` on npm) is a pure TypeScript library for building GraphQL schemas with full type safety, without code generation. Types are inferred directly from schema definitions, and a single schema definition powers both client query building and server resolver typing. `graphql` is the sole peer dependency.

## Commands

- `npm test` — runs Vitest with type-checking enabled (`vitest --typecheck --run`). Type-level tests in `*.test-d.ts` files are part of the test suite, so type errors are test failures.
- `npm run build` — builds the library via Vite. Multiple entry points (main + 3 integrations) output ESM (`.js`) and CJS (`.cjs`) to `dist/`, with `.d.ts` files generated by `vite-plugin-dts`.
- `npm run typecheck` — `tsc --noEmit` against the repo-root `tsconfig.json`.
- Run a single test file: `npx vitest run lib/builder.test.ts` (add `--typecheck` to include type tests).
- Docs site: `npm --prefix docs run dev` (fumadocs/Next.js, served locally).
- Demo app: `npm --prefix examples/demo run dev` (Next.js + urql + graphql-yoga).

## Publishing

The package is published as `@overstacked/typograph`. The `files` field ships only `dist/`. Subpath exports are configured for the main entry and three integrations (`./integrations/urql`, `./integrations/apollo`, `./integrations/react-query`), each with `types`, `import`, and `require` conditions.

## Architecture

The library is composed of three layers that share a single source of truth (the schema object built via `createTypeDefBuilder`):

1. **Schema builder (`lib/builder.ts`)** — `createTypeDefBuilder()` returns a builder with `type`, `query`, `mutation`, `subscription`, `inputType`, `field`, `typeDef`, and `combineTypeDefs` methods. At runtime most of these are pass-through identity functions; the heavy lifting happens in TypeScript's type system. `inputType(fields)` and `field({ input, output })` produce tagged wrappers (`{ __kind: "input", fields }` and `{ __kind: "field", input, output }`) that the SDL pipeline detects later. The exported `t` object provides scalar constructors (`t.string()`, `t.int()`, `t.boolean()`, `t.id()`) that return `TypeBuilder` instances. Scalar constructors also accept `{ default }` to attach a default value; the static type becomes `T | undefined` so variables become optional from the caller's perspective. `TypeBuilder` carries a runtime `typeString` (e.g. `"String"`), an `_isNonNull` flag, and an optional default, and overrides `toString`/`valueOf`/`Symbol.toPrimitive`/`toJSON` so it can be coerced to a GraphQL type string. `t.type<T>("Foo")` is the escape hatch used to reference named types (and is critical for circular references — wrap field references in arrow functions, e.g. `author: () => t.type<User>("User")`).

2. **SDL generation pipeline (`combineTypeDefs(...)`)** — runs three sequential passes over the combined typeDef object. The merge + evaluate steps run **eagerly** when `combineTypeDefs` is called, so `typeDefs.types` actually contains the merged + evaluated tree at runtime (not the raw input array). `toSDL()` only runs the final AST/print step on the cached evaluated tree.
   - `lib/utils/merge.ts` deep-merges all typeDefs (later values override earlier ones; nested objects are recursively merged).
   - `lib/utils/evaluate.ts` walks the merged object and (a) calls `.toString()` on any `TypeBuilder` instances (detected via `instanceof`, promoting builders with a default to a `{ type, default }` object instead of a bare string), (b) invokes any function-valued fields (this is what unwraps the lazy `() => t.type<...>(...)` references used for circular types), and (c) recurses into plain objects via `lib/utils/isObject.ts`.
   - `lib/utils/builderToAst.ts` converts the resulting plain object into a GraphQL `DocumentNode`. Top-level keys named `Query`, `Mutation`, or `Subscription` become operation fields (with `input` → arguments, `output` → return type). Keys whose value carries `__kind: "input"` become `INPUT_OBJECT_TYPE_DEFINITION`s; keys whose object fields carry `__kind: "field"` become `FIELD_DEFINITION`s with arguments; everything else becomes a plain `ObjectTypeDefinition`. Empty operation maps are skipped so we never emit `type Mutation {}`. The AST is then handed to `graphql.print` to produce SDL. Non-null types are detected by a trailing `!` in the type string.

3. **Type-safe client (`lib/client.ts`)** — `createClient(typeDefs)` returns `{ query, mutate, subscribe }`. It takes the *runtime* `typeDefs` object (the result of `combineTypeDefs(...)`) so that the operation handler can look up each operation's `input` map (variable name → GraphQL type string, possibly wrapped as `{ type, default }`) and emit proper `query Name($id: String) { name(id: $id) { ... } }` syntax. Each handler is an `OperationHandler` whose generic parameters extract the operation key, infer the response type from `Schema[Operation][Key]["output"]`, constrain the user-supplied selection set via `SelectionSet<T>` (boolean leaves for primitives, nested objects for object/array fields), and then map the selection back to a typed `returnType` via `SelectFields`. Subscriptions flow through the same handler (`GetOperationMap<Schema, "Subscription">` uses `NonNullable` to strip the optional-branch, so the Subscription map is treated uniformly). The returned object exposes `returnType` (phantom value, only the *type* matters — used as `typeof res.returnType` to recover the typed response shape), `variables` (the actual runtime values typed as the operation's input), and `toGraphQL()` which calls `lib/utils/buildGraphQLQuery.ts`. `lib/types/resolvers.ts` mirrors this for the server side, deriving a `Resolvers<T>` type with per-operation argument typing, per-type field resolvers, and `subscribe`/`resolve` pairs for Subscription fields.

### The `args(...)` helper

Exported from `lib/main.ts`, `args(argMap, selection)` wraps a selection with an explicit argument binding. It solves two problems:

1. **Nested field arguments.** Declare the field with `builder.field({ input, output })` on the parent type, then pass `args({ limit: "$limit" }, {...})` at the call site to emit `comments(limit: $limit) { ... }` and add `$limit` to the operation header.
2. **Top-level variable renaming.** Wrap a top-level operation with `args({ id: "$postId" }, {...})` so two operations sharing an arg name (e.g. two `id`s) can be disambiguated in the variables object.

The `ArgsWrapper<S, A>` runtime tag is unwrapped in both the client (`cleanSelections`, `argRenameByOperation`, `extraHeaderVarsByOperation`) and the type system (`ExtractSelection`, `RenameInputArgs`, `StripDollar`). The variable names in the arg map must start with `$` — the runtime strips exactly one leading `$` to derive the bare key, which is also what `buildGraphQLQuery` uses when emitting the header.

### Type-system conventions worth knowing

- The schema object is the single source of truth: `export type TypeDefs = typeof typeDefs;` is the bridge from runtime to types, consumed by both `createClient(typeDefs)` and `Resolvers<TypeDefs>`.
- Output type references in `t.type<T>("...")` use the GraphQL syntax in the runtime string (e.g. `"[Post]"`, `"User!"`) but the TypeScript generic for the static type (`Post[]`, `User`).
- Circular references between types **must** use a thunk: `comments: () => t.type<Comment[]>("[Comment]")`. `evaluate.ts` is what makes this work at runtime; on the type side, `ExtractValue` in `lib/types/common.ts` unwraps the function's return type.
- `lib/types/common.ts` defines `Prettify`, `Merge` (via `UnionToIntersection`), `BaseTypeDefs`, `FieldWithArgs`, and `PeelFieldArgs` — these are reused across `client.ts`, `resolvers.ts`, and `builder.ts` to keep inferred types readable and to enforce the `{ Query, Mutation, Subscription?, [type]: ... }` shape. `Mutation` is required for backwards compat with every existing typeDef that already declares `Mutation: {}`; `Subscription` is optional.

## Framework integrations

The library ships framework bridges under `lib/integrations/`, exposed as subpath exports (e.g. `@overstacked/typograph/integrations/urql`). Each integration is a factory that takes a typograph `typeDefs` object and closes over `createClient(typeDefs)` so the transport-level hooks can emit fully-typed selections without codegen.

`lib/integrations/urql.ts` exports `createUrqlIntegration(typeDefs)`, which returns `{ useQuery, useMutation, useSubscription }` bound to urql's React hooks. Two things to know when touching this file:

- `useMutation` deliberately bypasses urql's own `useMutation` hook and calls `client.mutation(...)` inside an `execute` callback. urql freezes the query string at hook-init time, but typograph only declares variables that are present when `toGraphQL()` runs — if the hook initialized with no variables, `$title`/`$body` would never appear in the operation header and would be silently dropped at execute time. Rebuilding the query inside `execute` keeps the header in sync with the real call's variables.
- The internal call sites cast `options` through `any` at the `typograph.query`/`typograph.mutate`/`typograph.subscribe` boundary. The hook-level `MergedVariables<Schema, Op, S>` type doesn't structurally satisfy typograph's inner `Variables` constraint (the `Prettify<MakeOptional<...>>` chain defeats assignability across generic boundaries). The public hook signatures are the real type gate, so external callers still get fully-checked inputs and outputs.

## Testing

- Tests live next to the source in `lib/`. There are two kinds:
  - Runtime tests: `*.test.ts` (Vitest).
  - Type-level tests: `*.test-d.ts` (e.g. `builder.test-d.ts`, `client.test-d.ts`, `full-stack.test-d.ts`, `types/resolvers.test-d.ts`). These exercise inference end-to-end and only fail when `--typecheck` is enabled, which is why `npm test` always passes that flag.
- `lib/examples.test.ts` is the end-to-end coverage file for every operation shape. Each example has its own `describe` block with three sub-tests: SDL string, execution against `graphql.execute` (or `graphql.subscribe`), and a typed-`returnType` assertion. When adding a new feature, extend this file with a corresponding example.
- `@graphql-tools/schema` is deliberately *not* used in the test rig — its transitive `graphql` import collides with our own under Vitest's SSR loader ("Duplicate graphql modules"). `examples.test.ts` uses `buildSchema` from `graphql` plus a tiny `attachResolvers` helper that bridges typograph's resolver signature (`(args)` for root, `(parent)` for type fields) to graphql.execute's `(source, args, ctx, info)`.
- When changing anything in `builder.ts`, `client.ts`, `types/`, or the `evaluate`/`merge`/`builderToAst` pipeline, run the full test suite — a regression usually shows up as a type error in `*.test-d.ts` rather than a runtime failure.
